2.3特殊字符

2.3.1空白字符

除了行终止符序列，ASCII水平空格字符（0x20）是唯一出现在源文件中任何地方的空格字符。这意味着

  1. 字符串文字中的所有其他空白字符都被转义，

  2. 标签字符不用于缩进。

2.3.2特殊转义序列

对于具有特殊的转义序列（任何字符\'，\"，\\，\b， \f，\n，\r，\t，\v），该序列使用，而不是对应的数字逃逸（例如\x0a，\u000a或\u{a}）。传统的八进制转义从不使用。

对于其余的非ASCII字符，使用实际的Unicode字符（例如∞）或等效的十六进制或Unicode转义（例如\u221e），这取决于使代码更容易阅读和理解。

术语注意：换行被定义为将单个表达式分解成多行。

当换行时，第一行（每条连续行）之后的每一行从原始行缩进至少+4，除非它落在块缩进的规则下。

4.6.1垂直空白

出现一个空白行：

  1. 在类或对象文字中的连续方法之间
    1. 异常：对象文字中两个连续属性定义之间的空白行（它们之间没有其他代码）是可选的。根据需要使用这些空白行来创建字段的逻辑分组。
  2. 在方法体内，谨慎地创建语句的逻辑分组。功能体开始或结束处的空行不允许。
  3. 可选地，在类或对象文字中的第一个或之后的最后一个方法（既不鼓励也不鼓励）。
  4. 根据本文其他部分的要求（例如 3.4 goog.require语句）。

4.6.2水平空白

使用水平空格取决于位置，并且分为三大类：领先（一行开始），尾随（在行尾）和内部。领先的空白（即缩进）在其他地方得到解决。尾随空白被禁止。

除了语言或其他风格规则所要求之外，除了文字，评论和JSDoc之外，单个内部ASCII空间也仅在以下位置出现。

  1. 分离任何保留字（如if，for或catch）从一个开括号（(它后面在该行）。
  2. 将任何保留字（如else或catch）从}该行上的其后的大括号（）中分离出来。
  3. 在任何开放的大括号（{）之前，有两个例外：
    1. 在作为函数的第一个参数的对象文字或数组文字中的第一个元素（例如foo({a: [{c: d}]})）之前。
    2. 在模板扩展中，因为它被语言（例如abc${1 + 2}def）禁止。
  4. 在任何二进制或三元运算符的两边。
  5. 在逗号（,）或分号（;）之后。请注意，空格从来没有这些字符之前允许的。
  6. 在冒号（:）之后的对象文字中。
  7. 在双斜线（//）的两边开始行尾注释。这里允许多个空格，但不是必需的。
  8. 在一个开放的JSDoc注释字符和两边的紧密字符（例如对于短格式类型声明或转换：this.foo = /** @type {number} */ (bar);或function(/** string */ foo) {）。

8.2.1使用标准警告集

8.2.2如何处理警告

在做任何事情之前，请确保你明确地了解警告告诉你什么。如果您不确定为什么出现警告，请寻求帮助。

一旦了解警告，请按顺序尝试以下解决方案：

  1. 首先，解决它或解决它。强烈地尝试解决警告，或者找到另一种方式来完成避免这种情况的任务。
  2. 否则，确定它是否是一个假警报。如果您确信警告无效，并且代码实际上是安全正确的，请添加评论以说服读者了解此事实并应用@suppress 注释。
  3. 否则，请留下TODO评论。这是最后的手段。如果这样做，不要压制警告。警告应该是可见的，直到它被妥善照顾。

8.2.3在最合理的范围内禁止发出警告

警告被压制在最窄的合理范围内，通常是单个局部变量或非常小的方法。通常，由于这个原因，提取变量或方法。

8.3弃用

8.4代码不在Google样式中

您偶尔会遇到您的代码库中不符合Google样式的文件。这些可能来自于收购，或者可能已经在Google Style在某些问题上发表了立场，或者由于其他任何原因可能在非Google Style中。

8.4.1重新格式化现有代码

更新现有代码的样式时，请遵循以下准则。

  1. 不需要更改所有现有的代码以满足当前的样式指南。重新格式化现有代码是代码流失和一致性之间的折衷。风格规则随着时间的推移而变化，这些调整以维持合规性将​​会造成不必要的流失。但是，如果对文件进行了重大更改，则预计文件将位于Google样式中。
  2. 小心不要让机会主义的风格修复混乱CL的焦点。如果您发现自己做了很多风格变化，对CL的中心焦点并不重要，请将这些更改推广到单独的CL。

8.4.2新添加的代码：使用Google样式

全新的文件使用Google样式，不管同一个包中其他文件的样式选择。

将新代码添加到不在Google样式中的文件时，建议重新格式化现有代码，但需遵循8.4.1重新格式化现有代码中的建议 。

如果没有完成重新格式化，则新代码应该与同一文件中的现有代码尽可能一致，但不得违反样式指南。

8.5本地风格规则

团队和项目可能会采用本文档以外的其他风格规则，但必须接受清理更改可能不符合这些附加规则，并且不得因违反任​​何其他规则而阻止此类清理更改。当心不要过分的规矩。风格指南不寻求在每一个可能的情况下定义风格，也不应该。

8.6生成代码：主要免除

由构建过程生成的源代码不需要在Google样式中。然而，任何由手写源代码引用的生成的标识符都必须遵循命名要求。作为一个特殊的例外，允许这样的标识符包含下划线，这可能有助于避免与手写标识符的冲突。

9.4.4一致性框架

JS Conformance框架是一个工具，它是Closure编译器的一部分，为开发人员提供了一种简单的方法来指定一组额外的检查，以便在标准检查上方的代码基础上运行。一致性检查可以例如禁止访问某个属性，或调用某个功能，或者丢失类型信息（未知数）。

这些规则通常用于执行关键限制（例如定义全局变量，可能会破坏代码库）和安全模式（例如使用 eval或分配到innerHTML）或更宽松地提高代码质量。
